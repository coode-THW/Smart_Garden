1.2 项目背景

智慧花园是一款基于计算机视觉技术 YOLOv11 的智能花卉养护助手，通过拍照识别花卉品种，提供科学、精准的养护建议。受服务器性能限制，系统采用手机端本地推理架构，将YOLO模型部署在移动设备上运行，无需后端服务器即可完成核心识别功能。

1.3 核心目标

  • 构建纯移动端的离线花卉识别能力，无需依赖后端服务器
  • 实现基于 YOLOv11 + ONNX Runtime 的本地推理方案
  • 设计非常见花卉的云端大模型识别增强方案
  • 本地化数据存储，保障用户隐私与离线可用性
  • 免注册即用：无需注册登录，打开APP即可拍照识别，降低首次使用门槛
  • 识别纠错闭环：用户可反馈识别错误，积累训练数据，驱动模型持续迭代

4. 核心模块详细设计

4.1 花卉识别模块

4.1.1 识别流程设计

┌──────────────────────────────────────────────────────────────────┐
│              花卉识别模块（全部在APP本地执行）                      │
├──────────────────────────────────────────────────────────────────┤
│  ┌──────────────┐      ┌──────────────┐      ┌──────────────┐  │
│  │ 图像预处理   │ ──→ │ YOLOv11推理  │ ──→ │  结果判断    │  │
│  │ (压缩/归一化)│      │ (ONNX本地)   │      │ (三级阈值)   │  │
│  └──────────────┘      └──────────────┘      └──────┬───────┘  │
│                    ┌────────────────────────────────┘          │
│                    ↓                                           │

│         │            │            │                           │

│  提示"未检测到   显示"联网识别   直接返回结果                    │

│  花朵拍摄"       │ 离线时提示                                │
│                 │ "网络不可用，                               │
│                 │  结果仅供参考"                              │
│                 └──→ 用户可选择"重新拍照"                     │
└──────────────────────────────────────────────────────────────────┘
识别结果页底部：
  ┌──────────────────────────────────────────────┐
  │  [根据养护指南一键设置提醒]  [识别有误？纠错]   │
  └──────────────────────────────────────────────┘
三级阈值说明
置信度区间判定行为
conf < 30%非花卉或图片质量过差拒绝识别，引导用户对准花朵重新拍摄，不调用 LLM
30% ≤ conf < 85%可能是花卉但不确定显示"正在联网识别..."，调 LLM 增强；无网络时返回低置信度结果 + 提示
conf ≥ 85%确认花卉本地直接返回结果
非花卉过滤的设计理由：用户可能误拍桌子、墙壁等非目标物体，模型强行匹配会输出荒谬结果。低于 30% 置信度时，LLM 也无法有效识别，调用 LLM 是浪费流量。明确拒绝 + 引导重拍，用户体验优于得到一个乱猜的名字。

4.1.2 启动预加载策略

为避免用户点击拍照按钮后才开始加载模型（白屏等待 2 秒），采用启动时异步预加载：

APP 启动
    ├──→ 显示欢迎页 / 主界面
    └──→ 异步加载 ONNX 模型（后台线程）
              ├── 加载中：拍照按钮显示"准备中..."（灰色，不可点击）
              │           或显示进度提示"识别引擎准备中..."
              └── 加载完成：拍照按钮激活（<2s，通常用户还没点）
                            如超过 2s 未完成，用户点击时继续等待并显示进度

预加载时机：APP 冷启动时，不与用户操作争抢资源。模型加载完成后常驻内存，后续识别无需重新加载。

4.1.3 识别纠错反馈

用户可纠正错误的识别结果，形成训练数据闭环：

识别结果页 ──→ 点击"识别有误？"
    │
    ↓
弹出输入框：输入正确花名（选填）或选择"不是花卉"
    │
    ↓
记录到本地日志：{ imageHash, yoloResult, confidence, userCorrection, timestamp }
    │
    ├── Phase 1：存储在本地 SQLite，可导出
    └── Phase 3（有服务器后）：上传至服务器
         → 人工审核 → 加入训练数据集 → 模型迭代
设计要点：纠错入口低调（底部文字链接），不打断用户流程。短期价值在积累数据，长期价值在模型迭代。

4.1.4 模型部署方案

YOLOv11模型通过ONNX Runtime直接在手机上运行推理。训练好的PyTorch模型导出为ONNX格式，经量化压缩后随APP打包分发。

部署方式适用场景优点缺点阶段
移动端本地推理当前方案无需网络，隐私保护，零服务器成本模型需压缩，推理速度受手机性能影响✅ 当前
NPU加速推理高端机型推理速度更快，功耗更低需适配各平台NPU（CoreML/NNAPI），开发量较大🔮 未来

4.1.5 YOLOv11模型配置

model:
  name: "yolov11n.onnx"
  input_size: 640
  confidence_threshold: 0.3       # 降低内部阈值以便捕获低置信度输出
  iou_threshold: 0.45
  max_det: 100

recognition:
  high_confidence: 0.85            # ≥85%：确认花卉，本地直接返回
  uncertain_range: [0.3, 0.85)     # 30%~85%：疑似花卉，调LLM增强
  reject_below: 0.3                # <30%或无检测结果：非花卉，拒绝并提示重拍

training:
  dataset: "custom_flower_dataset"
  epochs: 100
  batch_size: 32
  learning_rate: 0.001

export:
  format: "onnx";  opset: 17;  simplify: true

quantization:
  precision: "fp16"              # 半精度浮点，体积减半，精度损失极小
  alternative: "int8"            # 8位量化备选，体积更小但需校准数据集

deployment:
  platform: "mobile"
  runtime: "ONNX Runtime Mobile"
  providers:
    - "CPUExecutionProvider"     # 通用CPU推理（所有机型）
    - "CoreMLExecutionProvider"  # iOS NPU加速（iPhone 11+）
    - "NNAPIExecutionProvider"   # Android NPU加速（Android 10+）

4.2 养护知识数据策略

知识模块采用动静分离的混合数据策略：YOLO已训练花卉使用APP内置静态数据，大模型识别的非常见花卉由LLM动态生成。

识别结果 source 字段
        │
   ┌────┴────┐
   ↓         ↓
 yolov11    llm
   │         │
   ↓         ↓
flowerId 已知   flowerId 未知（不在训练集内）
   │         │
   ↓         ↓
查询本地静态养护库   识别时LLM已一并生成
(APP内置JSON)      careGuide 文本
   │         │
   ↓         ↓
返回完整结构化   返回LLM生成的养护建议
养护指南       （文本形式，结构略有差异）
数据来源覆盖范围存储方式内容特点
静态预置YOLOv11训练覆盖的常见花卉（~50+种）APP内置JSON资源文件，随应用打包结构化字段，人工校对，内容精准
LLM动态生成非常见花卉、稀有品种不存储，随识别结果一并返回文本形式养护建议

静态数据管理方式：花卉养护知识以JSON文件形式存放在APP资源目录中（如 static/care/月季.json），随APP打包分发作为基础数据。同时，APP 启动时自动检查远程知识库版本（Nginx 静态文件），有更新时增量下载，不依赖 APP 发版即可更新养护数据。

设计要点：服务器端只需 Nginx 目录下放 JSON 文件，零服务端逻辑、零额外进程。version.json 仅几十字节，增量下载通常 <50KB。服务器不可用时使用本地内置版本，不影响核心功能。

5. 模块间数据协议

架构说明：由于没有后端服务器，原RESTful API接口改为APP内部模块间的调用协议。以下定义保留接口的数据结构规范，供模块间调用时参考。各模块通过JavaScript函数调用直接交互，图片传递使用本地文件路径（无需Base64编码）。

5.1 通用数据规范

5.1.1 调用方式

  • 模块间调用：直接函数调用，同步或Promise异步
  • 大模型调用：uni.request HTTPS POST 直连云端API
  • 数据存储：SQLite（结构化用户数据）、本地JSON文件（知识库）、内存缓存（识别结果）

5.1.2 通用响应结构

{
    "code": 0,
    "message": "success",
    "data": { /* 业务数据 */ }
}

错误码含义说明

1002图片格式不支持仅支持JPG/PNG
1003图片尺寸超限最大2048×2048
2001识别失败模型推理异常
2002模型未加载ONNX模型加载失败
2003未检测到花卉置信度<30%，非花卉或图片质量过差，引导重拍

6. 非常见花卉识别方案（大模型云端增强）

6.1 设计背景

YOLOv11模型主要覆盖50+种常见家庭花卉，但对于稀有花卉、新品种或形态不典型的花卉，识别准确率可能不足。为此采用三级阈值策略：置信度 <30% 时判定为非花卉，直接拒绝并引导重拍；30% ≤ conf < 85% 时判定为疑似花卉，APP 通过 HTTPS 直连云端大模型进行二次识别；conf ≥ 85% 时确认花卉，本地直接返回。

6.2 识别编排器设计

识别编排器（Recognition Orchestrator）作为APP内的一个核心JS模块，负责统一调度花卉识别的完整流程。

职责说明
输入校验图片格式、尺寸、大小检查（本地路径直接读取）
本地缓存查询以图片哈希为key查询内存缓存，命中则直接返回
YOLO本地推理调用ONNX Runtime进行本地初步识别
置信度决策三级阈值：<30%拒绝+引导重拍；30%~85%触发LLM增强；≥85%本地返回
大模型API调用构造识别prompt，通过HTTPS直连云端大模型API
结果归一化无论来自YOLO还是大模型，统一封装为协议规范定义的格式
结果缓存回写内存缓存（图片哈希→识别结果），APP生命周期内有效
降级兜底大模型调用失败时，返回YOLO的低置信度结果+"仅供参考"提示

10. Phase 3 演进计划：轻量服务器补充

📅 计划阶段：Phase 3（Phase 1 + Phase 2 稳定运行后启动）

此章节为远期规划，用最小服务器成本补齐远程推送和模型迭代数据收集两个能力缺口。

⚠️ 知识库在线更新已提前至 Phase 1：只需 Nginx 静态文件托管，无需服务端进程。

10.1 设计原则

  • 最小化：不在服务器上跑任何重计算（YOLO推理、大模型调用仍在手机端和第三方API）
  • 轻量级：不用Spring Boot/MySQL/Redis，选Node.js+SQLite或Go嵌入式方案
  • 单机即可：2核4G足够撑几万用户
  • 可降级：服务器挂了不影响APP核心功能

10.3 远程推送

解决的问题：Phase 2 的养护提醒依赖APP本地通知，进程被系统杀死后通知失效。

   │                   │───────────────────→│

推荐：直接接个推或极光推送。服务端只需调一次 HTTP API，由推送平台负责送达。

10.4 识别纠错数据收集

Phase 1/2中用户纠错数据存储在本地SQLite。Phase 3通过WiFi下静默上传至服务器，经人工审核后加入训练数据集。

隐私保护：照片本身不上传，仅上传图片哈希+识别结果+用户纠正。上传为可选+WiFi下执行。
