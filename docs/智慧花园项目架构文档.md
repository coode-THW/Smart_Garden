## 1. 文档概述

### 1.1 文档信息
| 属性 | 内容 |
|------|------|
| **文档版本** | v2.2 |
| **创建日期** | 2026-06-10 |

| **作者** | 技术团队 |

### 1.2 项目背景
智慧花园是一款基于计算机视觉技术YOLOv11的智能花卉养护助手，通过拍照识别花卉品种，提供科学、精准的养护建议。受服务器性能限制，系统采用**手机端本地推理**架构，将YOLO模型部署在移动设备上运行，无需后端服务器即可完成核心识别功能。

### 1.3 核心目标
- 构建纯移动端的离线花卉识别能力，无需依赖后端服务器
- 实现基于YOLOv11 + ONNX Runtime的本地推理方案
- 设计非常见花卉的云端大模型识别增强方案
- 本地化数据存储，保障用户隐私与离线可用性
- **免注册即用**：无需注册登录，打开APP即可拍照识别，降低首次使用门槛
- **识别纠错闭环**：用户可反馈识别错误，积累训练数据，驱动模型持续迭代

| 层级 | 名称 | 职责 | 技术栈 |

| 分类 | 技术 | 版本 | 选型理由 |
|------|------|------|----------|
| **主识别模型** | YOLOv11n | 2024 | YOLO nano版本，模型小（~5MB），适合移动端 |
| **模型框架** | PyTorch → ONNX | 2.2+ | 训练用PyTorch，导出ONNX供移动端推理 |
| **移动端推理** | ONNX Runtime Mobile | 1.18+ | 跨平台移动推理引擎，支持CPU/NPU加速 |
| **模型量化** | FP16 / INT8 | - | 压缩模型体积，提升推理速度 |
| **大模型接口** | 阿里云百炼 / 智谱AI | - | 国内直连，图像理解能力强，成本低 |
| **大模型调用** | HTTPS 直连 (fetch API) | - | APP直接调用云端大模型API，无需中转服务器 |

### 3.3 与原云端架构的对比

| 维度 | 原方案（云端部署） | 当前方案（手机端本地） |
|------|-------------------|------------------------|
| **YOLO推理** | GPU服务器 | 手机CPU/NPU (ONNX Runtime) |
| **业务逻辑** | Spring Boot后端 | APP内JavaScript模块 |
| **数据存储** | MySQL + Redis | SQLite + 本地JSON |
| **接口调用** | HTTP RESTful API | APP内部函数调用 |
| **LLM调用** | 后端中转 | APP直连云端API |
| **网络依赖** | 每次识别需联网 | 低置信度时联网 + 知识库后台静默更新（可降级离线） |
| **运维成本** | 需维护服务器集群 | Phase 1/2 零服务器；Phase 3 可选轻量服务器（已有，复用） |

## 4. 核心模块详细设计

### 4.1 花卉识别模块

#### 4.1.1 识别流程设计

```
┌──────────────────────────────────────────────────────────────────┐
│              花卉识别模块（全部在APP本地执行）                      │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ┌──────────────┐      ┌──────────────┐      ┌──────────────┐  │
│  │ 图像预处理   │ ──→ │ YOLOv11推理  │ ──→ │  结果判断    │  │
│  │ (压缩/归一化)│      │ (ONNX本地)   │      │ (三级阈值)   │  │
│  └──────────────┘      └──────────────┘      └──────┬───────┘  │
│                                                     │          │
│                    ┌────────────────────────────────┘          │
│                    ↓                                           │
│              ┌───────────────┐                                │
│              │  置信度判断    │                                │
│              └───────┬───────┘                                │
│                      │                                        │
│         ┌────────────┼────────────┐                           │
│         ↓            ↓            ↓                           │
│   置信度 < 30%  30% ≤ conf < 85%  置信度 ≥ 85%                │
│   (非花卉/模糊)  (疑似花卉)       (确认花卉)                    │
│         │            │            │                           │
│         ↓            ↓            ↓                           │
│  提示"未检测到   显示"联网识别   直接返回结果                    │
│  花卉，请对准   中..."→调LLM     (本地)                        │
│  花朵拍摄"       │ 离线时提示                                │
│                 │ "网络不可用，                               │
│                 │  结果仅供参考"                              │
│                 │                                            │
│                 └──→ 用户可选择"重新拍照"                     │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘

识别结果页底部：
  ┌──────────────────────────────────────────────┐
  │  [根据养护指南一键设置提醒]  [识别有误？纠错]   │
  └──────────────────────────────────────────────┘
```

**三级阈值说明**：

| 置信度区间 | 判定 | 行为 |
|------------|------|------|
| **conf < 30%** | 非花卉或图片质量过差 | 拒绝识别，引导用户对准花朵重新拍摄，不调用 LLM |
| **30% ≤ conf < 85%** | 可能是花卉但不确定 | 显示"正在联网识别..."，调 LLM 增强；无网络时返回低置信度结果 + 提示 |
| **conf ≥ 85%** | 确认花卉 | 本地直接返回结果 |

**非花卉过滤的设计理由**：
- 用户可能误拍桌子、墙壁等非目标物体，模型强行匹配会输出荒谬结果
- 低于 30% 置信度时，LLM 也无法有效识别（图片本身缺乏花卉特征），调用 LLM 是浪费流量
- 明确拒绝 + 引导重拍，用户体验优于得到一个乱猜的名字

#### 4.1.2 启动预加载策略

为避免用户点击拍照按钮后才开始加载模型（白屏等待 2 秒），采用启动时异步预加载：

```
APP 启动
    │
    ├──→ 显示欢迎页 / 主界面
    │
    └──→ 异步加载 ONNX 模型（后台线程）
              │
              ├── 加载中：拍照按钮显示"准备中..."（灰色，不可点击）
              │           或显示进度提示"识别引擎准备中..."
              │
              └── 加载完成：拍照按钮激活（<2s，通常用户还没点）
                            如超过 2s 未完成，用户点击时继续等待并显示进度
```

**预加载时机**：APP 冷启动时，不与用户操作争抢资源。模型加载完成后常驻内存，后续识别无需重新加载。

#### 4.1.3 识别纠错反馈

用户可纠正错误的识别结果，形成训练数据闭环：

```
识别结果页 ──→ 点击"识别有误？"
    │
    ↓
弹出输入框：输入正确花名（选填）或选择"不是花卉"
    │
    ↓
记录到本地日志：
  { imageHash, yoloResult, confidence, userCorrection, timestamp }
    │
    ├── Phase 1：存储在本地 SQLite，可导出
    │
    └── Phase 3（有服务器后）：上传至服务器
         → 人工审核 → 加入训练数据集 → 模型迭代
```

> **设计要点**：纠错入口低调（底部文字链接），不打断用户流程。短期价值在积累数据，长期价值在模型迭代。

#### 4.1.4 模型部署方案

YOLOv11模型通过ONNX Runtime直接在手机上运行推理。训练好的PyTorch模型导出为ONNX格式，经量化压缩后随APP打包分发。

| 部署方式 | 适用场景 | 优点 | 缺点 | 阶段 |
|----------|----------|------|------|------|
| **移动端本地推理** | 当前方案 | 无需网络，隐私保护，零服务器成本 | 模型需压缩，推理速度受手机性能影响 | ✅ 当前 |
| **NPU加速推理** | 高端机型 | 推理速度更快，功耗更低 | 需适配各平台NPU（CoreML/NNAPI），开发量较大 | 🔮 未来 |

**当前选择移动端本地推理的理由**：
- 服务器性能有限，无法承载GPU推理负载
- YOLOv11 nano模型导出ONNX后仅约5MB，不会显著增加APP包体积
- ONNX Runtime 官方提供 `onnxruntime-react-native` npm 包，React Native 零桥接集成
- 离线识别是核心体验，无需依赖网络即可完成常见花卉识别（50+品种）
- 模型更新通过APP版本更新完成，无需维护独立的模型服务

#### 4.1.5 YOLOv11模型配置

```yaml
model:
  name: "yolov11n.onnx"
  input_size: 640
  confidence_threshold: 0.3       # 降低内部阈值以便捕获低置信度输出
  iou_threshold: 0.45
  max_det: 100

recognition:
  high_confidence: 0.85            # ≥85%：确认花卉，本地直接返回
  uncertain_range: [0.3, 0.85)     # 30%~85%：疑似花卉，调LLM增强
  reject_below: 0.3                # <30%或无检测结果：非花卉，拒绝并提示重拍

training:
  dataset: "custom_flower_dataset"
  epochs: 100
  batch_size: 32
  learning_rate: 0.001

export:
  format: "onnx"
  opset: 17
  simplify: true

quantization:
  precision: "fp16"              # 半精度浮点，体积减半，精度损失极小
  alternative: "int8"            # 8位量化备选，体积更小但需校准数据集

deployment:
  platform: "mobile"
  runtime: "ONNX Runtime Mobile"
  providers:
    - "CPUExecutionProvider"     # 通用CPU推理（所有机型）
    - "CoreMLExecutionProvider"  # iOS NPU加速（iPhone 11+）
    - "NNAPIExecutionProvider"   # Android NPU加速（Android 10+）
```

### 4.2 养护知识数据策略

知识模块采用**动静分离**的混合数据策略：YOLO已训练花卉使用APP内置静态数据，大模型识别的非常见花卉由LLM动态生成。

```
识别结果 source 字段
        │
   ┌────┴────┐
   ↓         ↓
 yolov11    llm
   │         │
   ↓         ↓
flowerId 已知   flowerId 未知（不在训练集内）
   │         │
   ↓         ↓
查询本地静态养护库   识别时LLM已一并生成
(APP内置JSON)      careGuide 文本
   │         │
   ↓         ↓
返回完整结构化   返回LLM生成的养护建议
养护指南       （文本形式，结构略有差异）
```

| 数据来源 | 覆盖范围 | 存储方式 | 内容特点 |
|----------|----------|----------|----------|
| **静态预置** | YOLOv11训练覆盖的常见花卉（~50+种） | APP内置JSON资源文件，随应用打包 | 结构化字段（浇水/施肥/光照/病虫害），人工校对，内容精准 |
| **LLM动态生成** | 非常见花卉、稀有品种 | 不存储，随识别结果一并返回 | 文本形式养护建议，在识别prompt中要求LLM一并输出 |

**静态数据管理方式**：花卉养护知识以JSON文件形式存放在APP资源目录中（如 `assets/care/月季.json`），随APP打包分发作为基础数据。同时，APP 启动时自动检查远程知识库版本（Nginx 静态文件），有更新时增量下载，**不依赖 APP 发版即可更新养护数据**。

**知识库在线更新流程**（Phase 1 即实现）：

```
APP 启动（后台静默）
    │
    ↓
GET https://your-server/knowledge/version.json
    │
    ↓
返回 { "version": 12, "updatedFiles": ["铁线莲.json", "蝴蝶兰.json"] }
    │
    │ 本地 version < 12 ?
    │
    ├── 否 → 跳过
    │
    └── 是 → 下载变更文件 → 覆写本地 assets/care/ → 更新本地版本号
```

> **设计要点**：服务器端只需 Nginx 目录下放 JSON 文件，**零服务端逻辑、零额外进程**。version.json 仅几十字节，增量下载通常 <50KB。服务器不可用时使用本地内置版本，不影响核心功能。详见 [第10章 Phase 3 演进计划](#10-Phase-3-演进计划轻量服务器补充) 中的服务端配置（知识更新部分可在 Phase 1 提前部署，只需 Nginx 静态文件托管）。

**LLM生成数据的处理**：LLM返回的养护建议以 `careGuide` 字段（文本）嵌入在识别响应中。将高频出现的非常见花卉的LLM生成结果人工审核后沉淀到静态库中，通过知识库在线更新下发，逐步扩大静态覆盖范围，减少对 LLM 的依赖。

## 5. 模块间数据协议

> **架构说明**：由于没有后端服务器，原RESTful API接口改为APP内部模块间的调用协议。以下定义保留接口的数据结构规范，供模块间调用时参考。各模块通过JavaScript函数调用直接交互，图片传递使用本地文件路径（无需Base64编码）。

### 5.1 通用数据规范

#### 5.1.1 调用方式
- **模块间调用**：直接函数调用，同步或Promise异步
- **大模型调用**：`fetch` HTTPS POST 直连云端API
- **数据存储**：SQLite（结构化用户数据）、本地JSON文件（知识库）、内存缓存（识别结果）

#### 5.1.2 通用响应结构
```json
{
    "code": 0,
    "message": "success",
    "data": {
        // 业务数据
    }
}
```

## 6. 非常见花卉识别方案（大模型云端增强）

### 6.1 设计背景

YOLOv11模型主要覆盖50+种常见家庭花卉，但对于稀有花卉、新品种或形态不典型的花卉，识别准确率可能不足。为此采用三级阈值策略：置信度 <30% 时判定为非花卉，直接拒绝并引导重拍；30% ≤ conf < 85% 时判定为疑似花卉，APP 通过 HTTPS 直连云端大模型进行二次识别（此为大模型增强识别的主要使用场景）；conf ≥ 85% 时确认花卉，本地直接返回。在 Phase 1/2 阶段，除知识库在线更新（静默增量下载，可降级）外，大模型增强识别是系统中主要需要网络的功能模块。Phase 3 将引入远程推送和纠错数据上传，同样均为可降级——网络不可用时 APP 核心功能不受影响。

### 6.2 识别编排器设计

识别编排器（Recognition Orchestrator）作为APP内的一个核心JS模块，负责统一调度花卉识别的完整流程。UI层只与编排器交互，不直接调用YOLO或大模型。

**编排器职责**：

| 职责 | 说明 |
|------|------|
| **输入校验** | 图片格式、尺寸、大小检查（本地路径直接读取） |
| **本地缓存查询** | 以图片哈希为key查询内存缓存，命中则直接返回 |
| **YOLO本地推理** | 调用ONNX Runtime进行本地初步识别 |
| **置信度决策** | 三级阈值：<30%拒绝 + 引导重拍；30%~85%触发LLM增强；≥85%本地返回 |
| **大模型API调用** | 构造识别prompt，通过HTTPS直连云端大模型API |
| **结果归一化** | 无论来自YOLO还是大模型，统一封装为协议规范定义的格式 |
| **结果缓存** | 回写内存缓存（图片哈希→识别结果），APP生命周期内有效 |
| **降级兜底** | 大模型调用失败时，返回YOLO的低置信度结果并附带"仅供参考，建议联网获取更准结果"提示 |

**编排器内嵌于APP的优势**：
- 无中间服务层，减少网络跳数和延迟
- YOLO识别完全离线，响应速度快
- 大模型仅偶尔调用，网络消耗极低
- 用户隐私数据不经过任何第三方服务器

**代码模块结构**：
```
src/
├── screens/
│   └── RecognizeScreen.tsx          # 识别页面（UI层）
├── orchestrator/
│   └── RecognitionOrchestrator.ts   # 编排层（核心调度逻辑）
├── services/
│   ├── YoloService.ts               # ONNX Runtime 本地推理封装 (官方 npm 包)
│   ├── LlmService.ts                # 云端大模型API调用封装

├── database/
│   └── db.ts                        # SQLite 数据库初始化与操作
└── assets/
```

### 6.4 大模型选择与评估

| 模型 | 提供商 | 国内直连 | 图像理解 | 知识储备 | 单张成本 | 推荐度 |
|------|--------|----------|----------|----------|----------|--------|
| **Qwen-VL-Max** | 阿里云（百炼） | ✅ 无需翻墙 | 优秀 | 优秀 | 约¥0.005 | ⭐⭐⭐⭐⭐ |
| **GLM-4V** | 智谱AI | ✅ 无需翻墙 | 优秀 | 优秀 | 约¥0.005 | ⭐⭐⭐⭐⭐ |
| DeepSeek-VL2 | DeepSeek | ✅ 无需翻墙 | 良好 | 良好 | 约¥0.002 | ⭐⭐⭐⭐ |
| GPT-4V | OpenAI | ❌ 需翻墙 | 优秀 | 优秀 | $0.03（~¥0.21） | - |
| Claude 3 | Anthropic | ❌ 需翻墙 | 良好 | 优秀 | $0.015（~¥0.10） | - |

> **注**：GPT-4V 和 Claude 3 在花卉识别能力上与国产模型差距不大，但对国内项目存在三重障碍：需境外支付方式注册、API需翻墙访问、成本高 20-50 倍。故不作为推荐选项。

**推荐方案**：采用 **Qwen-VL-Max（阿里云百炼）** 作为主模型，**GLM-4V（智谱AI）** 作为备用，实现故障切换。

**推荐理由**：
- **国内直连**：无需代理，手机APP可直接调用
- **成本极低**：按 1000 次测试算，两张图片识别花费不到 10 元；新用户通常有百万 token 免费额度
- **OpenAI 兼容接口**：两个平台均提供与 OpenAI API 格式兼容的接口，切换成本几乎为零
- **注册方便**：国内手机号 + 实名认证即可开通，无需外币信用卡
- **花卉识别够用**：Qwen-VL 和 GLM-4V 的视觉理解能力已覆盖花卉品种识别、特征描述、养护建议生成等全部需求

### 6.5 大模型调用方式

大模型调用通过 `LlmService.ts` 模块在APP内以HTTPS直连方式完成，不经过任何中间服务器。

```
┌─────────────────────────────────────────────────────┐
│                  APP 客户端                          │
│                                                     │
│  RecognitionOrchestrator.ts                         │
│       │                                             │
│       │  30% ≤ confidence < 85%                     │
│       ↓                                             │
│  LlmService.ts.recognize(imagePath, prompt)         │
│       │                                             │
│       │  HTTPS POST (TLS 1.3)                       │
│       │  fetch()                                    │
│       ↓                                             │
│  ┌─────────────────────────┐                        │
│  │  阿里云百炼 / 智谱 API   │  ← 云端服务           │
│  │  (Qwen-VL / GLM-4V)    │                        │
│  └─────────────────────────┘                        │
│       │                                             │
│       ↓                                             │
│  解析响应 → 提取结构化数据 → 返回编排器              │
│                                                     │
└─────────────────────────────────────────────────────┘
```

**LlmService.ts 职责**：

| 职责 | 说明 |
|------|------|
| **图片编码** | 将本地图片读取为Base64（大模型API要求） |
| **Prompt 构造** | 将图片和识别指令拼装为LLM可理解的请求体 |
| **模型路由** | 主模型（Qwen-VL-Max）失败时自动切换备用模型（GLM-4V） |
| **响应解析** | 从LLM返回的自由文本中提取结构化字段（花名/学名/科属等） |
| **结果校验** | 检查LLM返回的字段完整性，异常时重试或降级 |

**API Key 管理**：
- API Key 不硬编码在源码中，通过环境变量或配置文件注入
- 发布时使用代码混淆，增加逆向提取难度
- 建议在大模型平台设置调用额度上限，防止Key泄露后被盗刷

## 10. Phase 3 演进计划：轻量服务器补充

> 📅 **计划阶段**：Phase 3（Phase 1 核心识别 + Phase 2 本地提醒稳定运行后启动）
>
> 此章节为**远期规划**，描述在现有手机端本地推理架构基础上，用最小服务器成本补齐远程推送和模型迭代数据收集两个能力缺口。
>
> ⚠️ **知识库在线更新已提前至 Phase 1**：只需 Nginx 静态文件托管，无需服务端进程。详见 [4.2节 养护知识数据策略](#42-养护知识数据策略)。

### 10.1 设计原则

- **最小化**：不在服务器上跑任何重计算（YOLO 推理、大模型调用仍在手机端和第三方 API）
- **轻量级**：不用 Spring Boot / MySQL / Redis，选 Node.js + SQLite 或 Go 嵌入式方案
- **单机即可**：2核4G 足够撑几万用户，不引入集群、容器编排等复杂度
- **可降级**：服务器挂了不影响 APP 核心功能（识别、本地知识查询、本地提醒、知识更新）

### 10.3 远程推送

**解决的问题**：Phase 2 的养护提醒依赖 APP 本地通知，进程被系统杀死后通知失效。加上远程推送后，即使 APP 没在运行也能准时收到提醒。

**交互流程**：

```
┌──────────┐         ┌──────────────┐         ┌──────────────┐
│   APP    │         │  轻量服务器   │         │  推送平台     │
│          │         │              │         │ (个推/极光)   │
└────┬─────┘         └──────┬───────┘         └──────┬───────┘
     │                      │                        │
     │ ① 注册设备Token       │                        │
     │ POST /api/push/register                       │
     │ {token, platform, remindTasks}                │
     │─────────────────────→│                        │
     │                      │ 存储 device_tokens 表   │
     │                      │                        │
     │            ═══════════ 时间流逝 ════════════   │
     │                      │                        │
     │                      │ ② 定时检查提醒任务      │
     │                      │ SELECT * FROM reminders│
     │                      │ WHERE time <= now()    │
     │                      │                        │
     │                      │ ③ 调用推送平台API       │
     │                      │──────────────────────→│
     │                      │                        │
     │                      │       ④ 送达通知        │
     │  ← ← ← ← ← ← ← ← ← ← ← ← ← ← ← ← ← ← ← ←  │
     │  "该给月季浇水了"     │                        │
```

**推送平台选型**：

| 方案 | 优点 | 缺点 | 适合场景 |
|------|------|------|----------|
| **个推 / 极光推送** | 国内送达率高，免费额度大，SDK成熟 | 有厂商绑定感 | ✅ 推荐，开发量最小 |
| **直接调 FCM/APNs** | 无第三方依赖 | 需维护两套API + 证书，FCM国内不稳定 | 海外市场 |
| **自建长连接** | 完全控制 | 开发量巨大，手机耗电 | ❌ 不推荐 |

> **推荐**：直接接**个推**或**极光推送**。服务端只需调一次 HTTP API，由推送平台负责送达。

### 10.4 识别纠错数据收集

**解决的问题**：Phase 1/2 中用户通过"识别有误？"提交的纠错数据存储在本地 SQLite 中，无法汇总分析。Phase 3 有了服务器后，用户可选择上传纠错数据，形成模型迭代的训练数据来源。

**交互流程**：

```
APP 本地（Phase 1/2 已记录）               Phase 3 服务器
┌──────────────────────┐              ┌──────────────────┐
│ feedback 表（SQLite） │              │ feedback 表       │
│ · imageHash          │              │ · imageHash       │
│ · yoloResult         │   批量上传    │ · yoloResult      │
│ · confidence         │─────────────→│ · confidence      │
│ · userCorrection     │  (WiFi下)    │ · userCorrection  │
│ · timestamp          │              │ · timestamp       │
└──────────────────────┘              │ · userId          │
                                      └────────┬─────────┘
                                               │
                                               ↓
                                          人工定期审核
                                               │
                                               ↓
                                          加入训练数据集
                                               │
                                               ↓
                                          模型迭代 → 发版
```

**设计要点**：
- 上传为**可选 + WiFi下静默执行**，不消耗移动流量
- 照片本身不上传（隐私考虑），仅上传图片哈希 + 识别结果 + 用户纠正
- 服务器端每收到一定数量（如100条）通知管理员审核

### 10.5 技术选型建议

| 选项 | 内存占用 | 开发语言 | 适合场景 |
|------|----------|----------|----------|
| **Node.js + Express + better-sqlite3** | ~80MB | JavaScript（与APP同语言） | ✅ 推荐，无学习成本 |
| **Go + Gin + SQLite** | ~15MB | Go | 追求极致性能/低内存 |
| **crontab + shell 脚本** | 0MB 常驻 | bash | 仅推送的最简方案（不适用于纠错API） |

| **Phase 1** | 本地 | 本地JSON | Nginx静态文件 | 本地SQLite（匿名） | 直连云端API | — | — | 本地记录 |
